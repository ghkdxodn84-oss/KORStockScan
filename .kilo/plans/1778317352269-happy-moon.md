# 시스템 에러탐지 및 처리 자동화 - 작업지시서

생성일: `2026-05-09T18:02:32+09:00`

## 1. 목적

KORStockScan의 주요 런타임 구성요소(`bot_main.py`, 데몬 스레드, cron job, 단독실행 리포트)의 장애를 **자동 탐지·분류·알림**하는 `report-only` 운영 감시축을 구축한다.

- 현재 `log_error()`는 파일에만 적재되고, cron wrapper는 `[FAIL]` 표시만 남긴다. 사람이 수동으로 로그를 확인하지 않으면 장애가 방치된다.
- Sentinel(매매 병목/청산 이상치)과 달리 **시스템 인프라 자체의 정상작동 여부**를 감시하는 축이 없다.

이 작업지시서는 `runtime_effect=false`인 구현 설계서다. runtime threshold/strategy/주문 판단을 변경하지 않는다.

## 2. 현재 상태 분석

### 2.1 기존 오류 처리 현황

| 계층 | 방식 | 문제 |
| --- | --- | --- |
| `bot_main.py` main loop | `except Exception: log_error + sleep 5` | 스레드 사일런트 다운 감지 못함 |
| 데몬 스레드(telegram/crisis/scalper) | 각 스레드 내부 개별 try/except | 스레드 종료 시 main에 보고 없음 |
| cron wrapper (`.sh`) | `set -euo pipefail`, flock lock, `[FAIL]` 로그 | 종료 후 아무도 읽지 않음 |
| `log_error()` | 모듈별 error 로그 파일 | 자동 스캔/집계/알림 없음 |
| `system_metric_sampler.py` | 1분 주기 CPU/메모리/loadavg/disk | metric jsonl만 쌓고 판정/알림 없음 |
| sentinel (buy_funnel, holding_exit) | 5분 주기 pipeline event 분석 | 매매/청산 병목 전용. 인프라 장애 미감지 |

## 3. 설계 결정

### 3.1 원칙

- **report-only**: runtime threshold, 주문, 청산 판단을 바꾸지 않는다.
- **자동 복구 제한**: 프로세스 재시작, 스레드 재생성, 잠금 파일 정리처럼 안전한 작업만 자동 수행. 매매/임계값/설정 변경은 금지.
- **확장성**: 새로운 기능 추가 시 탐지 대상으로 편입할 수 있도록 plugin registry 패턴 사용.
- **runbook과 정합**: 판정 상태(`pass`/`warning`/`fail`/`not_yet_due`)와 장애 대응 기준을 runbook과 일치시킨다.
- **기존 infra 재사용**: EventBus, log_error, TradingConfig, system_metric_sampler 데이터를 활용한다.

### 3.2 아키텍처

```
src/engine/error_detector.py          ← 중앙 엔진 + registry
src/engine/error_detectors/
    __init__.py                        ← 패키지 init

    process_health.py                  ← 메인루프/스레드 감지
```

## 4. Detector 상세 설계

### 4.1 ProcessHealthDetector (`process_health.py`)

**탐지 대상**: `bot_main.py` 메인루프와 모든 데몬 스레드

**구현 방식**:
- heartbeat 파일 (`tmp/error_detector_heartbeat.json`)을 각 감시 대상이 주기적으로 갱신
- `bot_main.py` main loop: 1초 루프에서 매 5회(5초)마다 heartbeat 갱신
- 데몬 스레드: 각 스레드 루프에서 매 30초마다 heartbeat 갱신 (기존 코드에 최소한의 instrumentation 추가)
- detector가 heartbeat 파일을 읽어 마지막 갱신 시각이 threshold 초과 시 `fail` 판정

**heartbeat schema**:
```json
{
  "main_loop": {"last_beat": "2026-05-09T18:02:00+09:00", "pid": 12345},
  "threads": {
    "telegram": {"last_beat": "...", "alive": true},
    "crisis_monitor": {"last_beat": "...", "alive": true},
    "sniper_engine": {"last_beat": "...", "alive": true},
    "scalping_scanner": {"last_beat": "...", "alive": true}
  }
}
```

**severity 판정**:
- `pass`: 모든 heartbeat < threshold
- `warning`: 1개 스레드 heartbeat stale (60~120초)
- `fail`: main loop heartbeat stale (>15초) 또는 스레드 heartbeat stale (>120초)

**auto_recovery**: `fail` 시 EventBus `SYSTEM_HEALTH_ALERT` 발행. 스레드 자동 재시작은 bot_main.py에 `restart_thread()` 핸들러 등록 필요.

### 4.2 CronCompletionDetector (`cron_completion.py`)

**탐지 대상**: runbook 시간표에 등록된 모든 cron job

**구현 방식**:
- runbook의 시간표(`docs/time-based-operations-runbook.md`)를 기준으로 `cron_job_registry` 정의
- 각 cron wrapper 출력 로그 파일에서 최근 `[START]`/`[DONE]`/`[FAIL]` 마커를 파싱
- 시간대별 기대 실행 구간을 벗어났는데 `[DONE]`이 없으면 `warning`/`fail`

**cron_job_registry 구조** (코드 내 상수 정의):
```python
CRON_JOB_REGISTRY = [
    {"id": "final_ensemble_scanner", "log": "logs/ensemble_scanner.log",
     "window_start": "07:20", "window_end": "08:00", "mode": "once"},
    {"id": "buy_funnel_sentinel", "log": "logs/run_buy_funnel_sentinel.log",
     "window_start": "09:05", "window_end": "15:20", "mode": "recurring", "interval_min": 5},
    ...
]
```

**severity 판정**:
- `pass`: 예정된 모든 cron이 정상 완료
- `warning`: 1개 cron 지연(아직 window 내) 또는 1회 실패로 retry 예정
- `fail`: 필수 cron 미실행 또는 반복 실패
- `not_yet_due`: 아직 실행 시간 도달 전

**auto_recovery**: 없음. `fail`은 Telegram 알림만.

### 4.3 LogScanner (`log_scanner.py`)

**탐지 대상**: `logs/` 디렉토리의 모든 error 로그 파일

**구현 방식**:
- state 파일(`tmp/error_detector_log_scan_state.json`)에 각 로그 파일의 마지막 스캔 위치 저장
- 스캔 시점 이후 새로 추가된 error line을 읽어 분류
- 오류 유형별로 집계 (exception type, 발생 모듈, 발생 빈도)

**분류 유형**:
- `DB_ERROR`: DB 연결/쿼리 실패
- `API_ERROR`: Kiwoom API, AI provider 호출 실패
- `WEBSOCKET_ERROR`: WS 연결 실패
- `PARSE_ERROR`: 데이터 파싱/스키마 실패
- `TIMEOUT_ERROR`: 연산/IO 타임아웃
- `UNKNOWN`: 분류되지 않은 예외

**severity 판정**:
- `pass`: 새 오류 없음
- `warning`: 5분간 1~3건 신규 오류
- `fail`: 5분간 4건 이상 또는 단일 유형 3건 이상

**auto_recovery**: 없음. 알림만.

### 4.4 ArtifactFreshnessDetector (`artifact_freshness.py`)

**탐지 대상**: 각 cron/데몬이 생성하는 중요 출력 파일

**구현 방식**:
- `artifact_registry`에 각 파일의 예상 경로, 생성 주기, 최대 허용 지연시간 정의
- 파일이 존재하고 mtime이 window 내인지 확인

**artifact_registry 예시**:
```python
ARTIFACT_REGISTRY = [
    {"path": "data/pipeline_events/pipeline_events_{date}.jsonl",
     "max_staleness_sec": 300, "mode": "append"},
    {"path": "data/threshold_cycle/runtime_env/threshold_runtime_env_{date}.json",
     "max_staleness_sec": 600, "mode": "once", "window": "07:35~08:00"},
    ...
]
```

**severity 판정**:
- `pass`: 모든 artifact 존재하고 fresh
- `warning`: 보조 artifact 누락
- `fail`: 필수 artifact(threshold env, pipeline events 등) 누락
- `not_yet_due`: window 도달 전

**auto_recovery**: 없음.

### 4.5 ResourceUsageDetector (`resource_usage.py`)

**탐지 대상**: 시스템 리소스 (CPU, 메모리, 디스크, swap)

**구현 방식**:
- 기존 `system_metric_sampler`의 `logs/system_metric_samples.jsonl`에서 최근 N개 샘플 읽기
- 직접 `/proc` 읽기 (sampler 없이도 동작 가능하도록 fallback)

**임계값** (TradingConfig에 추가):
```python
ERROR_DETECTOR_CPU_BUSY_MAX_PCT: float = 90.0
ERROR_DETECTOR_MEM_AVAILABLE_MIN_MB: float = 500.0
ERROR_DETECTOR_DISK_FREE_MIN_MB: float = 2048.0
ERROR_DETECTOR_SWAP_USED_MAX_PCT: float = 80.0
ERROR_DETECTOR_LOADAVG_15M_MAX: float = 8.0
```

**severity 판정**:
- `pass`: 모든 지표 정상
- `warning`: 1개 지표가 임계 근접(80% 수준)
- `fail`: 1개 지표 임계 초과

**auto_recovery**: disk full 시 `log_rotation_cleanup` trigger 가능 (별도 구현 필요). 기본은 알림만.

## 7. 판정 기준

- `runtime_effect=false`: 구현한 모든 detector는 observe-only. 매매 threshold 변경 없음.
- `target_subsystem`: `runtime_instrumentation`
- `lifecycle_stage`: `runtime_ops`
- `intent`: 계측 보강
- `expected_ev_effect`: 운영 장애 감지 시간 단축, 장애 방치로 인한 기회비용 손실 감소 (정량화는 postclose daily EV에서 RUNTIME_OPS 지표로 확인)

## 8. 확장성 설계

새로운 모듈이나 기능이 추가될 때 에러탐지 대상으로 편입하는 절차:

1. 신규 `detector_*.py` 파일을 `src/engine/error_detectors/`에 생성
2. `BaseDetector`를 상속하고 `check()` 구현
3. `@register_detector` 데코레이터로 registry에 등록
4. `TradingConfig`에 필요한 임계값 상수 추가
5. cron job인 경우 `CRON_JOB_REGISTRY`에 항목 추가
6. 출력 artifact인 경우 `ARTIFACT_REGISTRY`에 항목 추가
7. 심각도 판정은 runbook의 `pass/warning/fail/not_yet_due`를 따름

## 10. 운영 메모

- 탐지 결과 artifact: `data/report/error_detection/error_detection_YYYY-MM-DD.json`
- heartbeat state: `tmp/error_detector_heartbeat.json`
