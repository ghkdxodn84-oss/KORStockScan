# 2026-04-16 진입 후 수익전환 노력 공백 분석

작성일: 2026-04-16  
분석 범위: 현재 관찰축 1~5, 튜닝 계획 전체, 보유 중 종목 실적 데이터

---

## 1. 핵심 문제 정의

### 현재 AI가 보유 중에 하는 일

```
"이 종목 아직 들고 있어야 해?"
  → score 계산 → HOLD 또는 SELL 판단
```

### 수익전환을 위해 AI가 해야 할 일 (현재 미구현)

```
"지금 어떤 행동을 취하면 수익을 낼 수 있어?"
  → ADD / PYRAMID / 목표가 상향 / HOLD_TIGHT / FORCE_EXIT
```

**진입하고 기다리는 구조다.** 보유 중 수익전환을 위한 액티브한 행동 결정 구조가 없다.

이는 "동전 던지기보다 못하다"는 평가의 구조적 근거다.  
좋은 종목을 찾아도 들어간 뒤 어떻게 운영할지에 대한 두뇌가 없는 상태이기 때문이다.

## 2. 관찰축 1~5 수익전환 기여 맵

| 관찰축 | 목적 | 단계 | 수익전환 직접 기여 |
|--------|------|------|-------------------|
| 1. 퍼널 병목 (`submitted/filled` 분리) | 진입 체결률 개선 | 진입 전 | ❌ 없음 |
| 2. `partial fill sync mismatch` | 체결 정합성 확보 | 진입 직후 | ❌ 없음 |
| 3. `RELAX-DYNSTR momentum_tag` | 진입 필터 완화 | 진입 전 | ❌ 없음 |
| 4. `holding skip / gatekeeper reuse` 실패사유 | AI 레이턴시 병목 계측 | 보유 중 | △ 간접 기여만 |
| 5. `add_judgment_locked` blocker 분포 | 물타기 차단 계측 | 보유 중 | ❌ 물타기 OFF라 실효 없음 |

**관찰축 5개 전부 진입 품질 또는 레이턴시 개선 축. 수익전환 직접 기여 0.**

## 3. 현재 HOLDING AI의 구조적 결함

`2026-04-13-scalping-holding-prompt-final-design.md`에 이미 진단되어 있으나 미구현 상태다.

> **HOLDING은 "진입 판단 프롬프트를 HOLDING에서도 재사용하는 구조"다.**  
> 코드는 `SELL`을 보지만, 현행 스캘핑 프롬프트는 `BUY | WAIT | DROP`만 반환한다.

보유 중 AI에게 "지금 어떻게 할까"를 물어봐도 AI는 **"지금 살까 말까"** 의 프레임으로 대답한다.  
보유 특화 판단 구조가 없다.

### 3.1 action 불일치

| 경로 | 현재 반환 action | 본래 필요한 action |
|------|-----------------|-------------------|
| WATCHING (진입 판단) | `BUY / WAIT / DROP` | 유지 적절 |
| HOLDING (보유 판단) | `BUY / WAIT / DROP` (재사용 중) | `HOLD / SELL / FORCE_EXIT` |
| PRESET_TP (익절 검문) | 코드는 `SELL` 확인, 프롬프트는 `DROP` 반환 | `EXTEND / EXIT` |

### 3.2 입력 데이터 불일치

현재 `_format_market_data()`는 HOLDING 경로에도 아래를 그대로 주입한다:

- raw 분봉 시계열
- raw 호가창
- 최근 10틱 상세 원본

스캘핑 HOLDING 경로에는 과중하고, 포지션 컨텍스트(현재 손익·보유시간·고점 대비 되밀림)가 오히려 부족하다.

## 5. 오늘 보유 중 종목으로 확인한 증거

### 5.1 롯데쇼핑(023530) — 74분 보유, 수익전환 없음

```
보유: 118,800원 × 2주 (요청 11주 중 2주만 체결)
수익률 범위: -1.07% ~ -0.06%  (단 한 번도 breakeven 미달성)
AI 점수 범위: 31 ~ 71점  (방향성 없는 진동)
보유 중 취해진 액션: 없음 (waiting only)
```

**AI는 336회 리뷰하는 동안 "HOLD"만 반복. 수익 방향으로 포지션을 개선하는 시도 없음.**

### 5.2 올릭스(226950) — 36분 보유, +0.04% 달성 후 복귀

```
보유: 183,800원 × 1주 (요청 7주 중 1주만 체결)
수익률 범위: -0.50% ~ +0.04%
AI 점수: 62~80점 (양호 구간)
보유 중 취해진 액션: 없음
```

**+0.04% 순간 달성했으나 preset TP(고정)에 도달 못해 익절 없음. AI가 "지금 목표가를 조정해야 한다"는 판단을 할 구조가 없음.**

### 5.3 공통 패턴

```
진입 → AI 보유감시(score 계산) → HOLD 반복 → 손절선 도달 → 매도
                ↑
          이 구간에서 수익 방향으로 개선하는 메커니즘 없음
```

## 6. 수익전환 노력 보강 우선순위

### 6.1 즉시 적용 가능 — 코드 있음, 스위치만 켜면 됨

```python
REVERSAL_ADD_ENABLED = True
REVERSAL_ADD_MAX_HOLD_SEC = 300   # 120초 → 5분으로 확장
```

**단, 전제 조건**: `buy_qty >= 5` 이상 체결된 포지션에서만 실효성 있음.  
진입 체결률 개선이 병행되지 않으면 효과 제한적.

---

### 6.2 단기 — 설계 완료, 구현 필요 ★ 가장 높은 기대값

**HOLDING 전용 프롬프트 분리 구현**

현재 가장 큰 구조적 공백. 설계(`2026-04-13-scalping-holding-prompt-final-design.md`)는 이미 있다.

| 경로 | 역할 | action | 호출 조건 |
|------|------|--------|-----------|
| `HOLDING_GENERAL` | 일반 보유 유지 vs 청산 판단 | `HOLD / SELL / FORCE_EXIT` | 기존 리뷰 주기 |
| `HOLDING_CRITICAL` | 즉시 탈출 여부 판단 | `HOLD / SELL / FORCE_EXIT` | `profit_rate < 0` 또는 `>= 0.5%` |
| `PRESET_TP` | 익절 연장 vs 즉시 청산 | `EXTEND / EXIT` | `profit_rate >= 0.8%` 1회 검문 |

이 구조가 구현되면 보유 AI가 비로소 "이 포지션에서 수익을 낼 수 있는가"를 포지션 컨텍스트 기반으로 판단하게 된다.

**입력 개선 방향:**

```
현재: raw 분봉 + raw 호가 + 10틱 원본  (진입 판단 기준)
개선: 현재 손익률 + 보유시간 + 고점 대비 되밀림 + AI score 이력
     (보유 판단에 필요한 파생 수치만 주입)
```

---

### 6.3 중기 — 설계 필요

#### 동적 목표가 조정

현재 preset TP는 진입 시 고정 후 불변.  
롯데쇼핑처럼 74분 보유하는 동안 시장 상황이 변해도 목표가가 그대로다.

필요한 판단:
- `"모멘텀이 약해졌으니 목표가를 낮춰 조기 탈출"`
- `"AI 점수가 회복됐으니 목표가 유지 또는 상향"`

현재는 이 판단을 내리는 구조 자체가 없다.

#### SCANNER 포지션 보유 timeout 명시화

```python
# SCANNER 포지션: 30분 초과 + peak < 0% + AI < 55점 → 강제 청산 고려
SCANNER_MAX_HOLD_SEC_NEVER_GREEN = 1800
```

롯데쇼핑(74분), 올릭스(36분)처럼 SCANNER 포지션이 스캘핑 개념을 벗어나 장기 표류하는 패턴을 구조적으로 막을 기준이 없다.

## 7. 공백 구조 요약

```
진입 결정 (WATCHING AI)
    │
    ▼
체결 (partial fill 문제 → 관찰축 1~3에서 개선 중)
    │
    ▼
보유 감시 (HOLDING AI)  ← 현재 공백 구간
    │
    ├─ HOLD 반복 (수익전환 행동 없음)
    ├─ 물타기 / 피라미딩 OFF
    ├─ 동적 목표가 없음
    └─ HOLDING 전용 프롬프트 없음 (진입 프롬프트 재사용)
    │
    ▼
손절선 도달 → 매도
```

| 구분 | 현황 | 판단 | 검증 의견(코덱스) |
|------|------|------|-------------------|
| 진입 AI 품질 개선 | 관찰/튜닝 진행 중 | 개선 중 | **맞음.** 관찰축/튜닝이 entry·체결 품질 중심으로 운영되고 있으며 보유 전환 액션까지는 아직 확장되지 않음. |
| 보유 AI (수익전환 판단) | WATCHING 프롬프트 재사용, 전용 판단 없음 | **공백** | **부분보정 필요.** 완전 동일 재사용이라기보다 공통 시스템 프롬프트 + `task_type=scalping_holding` 분기 구조. 다만 전용 action schema(`HOLD/SELL/FORCE_EXIT`) 미분리라 실질 공백 진단은 타당. |
| 포지션 액션 (물타기/피라미딩) | 코드 있음, 전부 OFF | **공백** | **맞음.** `REVERSAL_ADD_ENABLED=False`, `SCALP_LOSS_FALLBACK_ENABLED=False`로 실전 액션이 비활성. |
| 청산 타이밍 최적화 (PRESET_TP) | Phase 3, 미착수 | **공백** | **대체로 맞음.** PRESET_TP 전용 의사결정(연장/즉시청산) 분리 구현이 아직 미완. |
| 동적 목표가 조정 | 설계 없음 | **공백** | **맞음.** 목표가 실시간 재설정 로직이 운영 경로에 없음. |
| SCANNER 포지션 timeout | 설계 없음 | **공백** | **맞음.** 장기 표류를 강제 정리하는 명시 timeout 정책이 부족하며 실거래 장시간 보유 케이스로 확인됨. |
