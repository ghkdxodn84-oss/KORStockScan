# 2026-04-10 Post-Tuning Data Platform Plan

## 목적

- 이 문서는 `튜닝 이후` 진행할 데이터 저장/조회 구조 개선안 전용 문서다.
- 본서비스 실매매 경로 영향 최소화와 운영 효율을 동시에 만족하는 방안을 정의한다.

## 판정

1. `S3 단독` 또는 `DB 단독`보다 `Hybrid`가 최적이다.
2. 권장안은 `로컬 JSONL(실매매 경로 유지) + 비동기 S3 적재 + 분석/서빙 DB 후처리`다.

## 옵션 비교 (실매매 영향 기준)

| 옵션 | 실매매 영향 | 운영 난이도 | 조회/분석 효율 | 판정 |
| --- | --- | --- | --- | --- |
| DB 직접쓰기(주문 경로 동기) | 높음 | 중간 | 높음 | 비권장 |
| S3 단독(파일 기반 조회) | 낮음 | 중간 | 낮음~중간 | 단기 가능 |
| Hybrid(JSONL + S3 + DB) | 낮음 | 중간~높음 | 높음 | 권장 |

## 권장 아키텍처

1. 실매매/의사결정 경로:
   - 기존 JSONL append 유지(`bot_main` 주문 경로 비차단).
2. 수집/보관 경로:
   - sidecar 또는 cron으로 `30~60초` 단위 S3 업로드(실패 재시도, 주문 경로 분리).
3. 분석/서빙 경로:
   - S3 원본을 배치 적재해 DB(리포트 전용) 구성.
   - 웹앱 API는 점진적으로 DB 조회로 전환.

## 단계별 실행 (튜닝 이후)

1. Phase 1 (`Safe Ingest`):
   - JSONL -> S3 비동기 적재만 추가, 기존 API 변경 없음.
2. Phase 2 (`Serve DB`):
   - S3 -> DB 적재 파이프라인 구축, 핵심 API 1~2개부터 DB로 전환.
3. Phase 3 (`Fallback Hardening`):
   - DB 조회 장애 시 JSONL/S3 fallback 경로 유지.
4. Phase 4 (`Retention Optimization`):
   - 로컬 JSONL 단기 보관, 장기 보관은 S3 중심으로 정리.

## 운영 가드레일

1. 실매매 경로에 네트워크 의존 동기 쓰기 금지.
2. 변경은 `읽기 경로`부터 단계적으로 전환하고, 쓰기 경로는 마지막에 평가.
3. 장애 시 즉시 기존 JSONL 기준으로 롤백 가능해야 한다.
4. 비교/손익 집계 기준(`COMPLETED + valid profit_rate`)은 기존 원칙을 그대로 유지한다.

## 체크포인트

1. 주문 제출 지연(`order submit latency`) 변화 0 또는 허용범위 내.
2. 누락 이벤트율 0에 수렴(재전송/중복제거 포함).
3. 대시보드 API p95 응답시간 개선.
4. 장중/장후 리포트 재현성 유지.
