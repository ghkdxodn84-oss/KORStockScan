## 문서 목적
- 이 문서는 기존 코드리뷰 결과를 **현재 코드 기준으로 재검증**하고,  
  **로직 불변(거래 의사결정 규칙 유지)**을 전제로 한 실행 계획을 정의한다.
- 본 문서 작성 턴에서는 **코드 수정 없이** 계획과 우선순위만 정리한다.

### 핵심 해석

1. 동기화 스레드 중복 실행 + 공유상태 동시 접근
2. 외부 I/O 실패의 무음 처리(silent failure)
3. 토큰/API 키 일부 로그 노출

## 3) 리스크 상세 및 보완 방향

### P0-1. 동시성/상태 정합성
- 관찰:
  - 메인 루프에서 90초마다 `periodic_account_sync`를 새 스레드로 생성.
  - 공유 상태(`ACTIVE_TARGETS`, `cooldowns`, `highest_prices`) 접근이 구간별로 락 일관성이 다름.
- 보완 방향:
  - `periodic_account_sync` 단일비행 제어(실행 중 재진입 금지).
  - 공유 상태 접근 래퍼 표준화(읽기/쓰기 공통 규칙).
  - 메인 루프/동기화 루프의 락 경계 정의 문서화.

### P0-2. 외부 API 실패의 무음 처리
- 관찰:
  - 일부 경로가 예외 후 무로그 기본값 반환(예: 예수금 조회 실패 시 `0`).
- 보완 방향:
  - “외부 I/O 실패”는 최소 `log_error + 원인코드`를 남기고, 정상값과 구분 가능한 상태값으로 처리.
  - fallback은 유지하되, 관측 가능성(알람/집계)을 동시에 확보.

### P0-3. 민감정보 로그 노출
- 관찰:
  - 키/토큰 일부(suffix/preview)가 로그로 남는 경로 존재.
- 보완 방향:
  - 민감정보는 전면 마스킹(`***`) 원칙.
  - 키/토큰 관련 로그 포맷 표준화 및 린트성 점검 추가.

### P1. 예외 정책 정밀화 (로직 불변 핵심)
- 관찰:
  - `except Exception` 전면 제거는 파서/정규화 fail-safe 로직까지 깨뜨릴 수 있음.
- 보완 방향:
  - 아래 매트릭스로 구분:
    - 파서/정규화 계층: 보수적 fallback 허용(단, 중요 이벤트는 카운터/로그).
    - 외부 I/O 계층: 실패 원인 로깅 + 재시도/쿨다운 정책.
    - 상태전이/주문 계층: 의도치 않은 예외는 fail-closed + 명시적 알림.

### P1. 테스트 보강
- 보완 방향:
  - 동시성(single-flight), 무음실패 제거, 민감정보 마스킹에 대한 회귀 테스트 추가.
  - 기존 핵심 테스트 세트(상태머신/주문/리포트)를 기준으로 변경 영향도 검증.

## 4) 실행 계획 (코드 수정 전제 아님, 단계별)

목표:

산출물:

4. 회귀 테스트 실행 목록(필수/선택) 확정

완료 기준:

## Phase 1. P0 최소 변경 묶음 (추후 코드 작업)
범위:
- 동기화 단일비행
- 무음 실패 경로 관측성 확보
- 민감정보 마스킹

가드레일:
- 매수/매도 판단 임계값, 게이트 조건, 청산 규칙은 변경 금지.
- DB 스키마 변경 금지.
- 외부 API 계약(입출력 필드) 변경 금지.

검증 기준:
- 기존 핵심 테스트 통과
- 장중 루프 오류율/재시도율 악화 없음
- 동기화 중복 실행 로그 0건
- 토큰/키 패턴 로그 0건

롤백 기준:
- 주문 상태전이 이상(중복 체결 처리/상태 고착) 징후 발생
- 성능 저하(루프 지연, API 실패율 급증) 확인 시 즉시 롤백

## 7) P1 작업오더 (반영본, 코드 수정 전 준비 단계)

### P1-WO-01. 예외 정책 매트릭스 확정 (로직 불변 기준선 고정)
- 목적:
  - `except Exception`를 기계적으로 제거하지 않고, 계층별 허용/금지 정책을 먼저 확정한다.
- 대상 파일(우선):
  - `src/engine/sniper_state_handlers.py`
  - `src/engine/sniper_sync.py`
  - `src/engine/kiwoom_sniper_v2.py`
  - `src/engine/kiwoom_orders.py`
  - `src/utils/kiwoom_utils.py`
  - `src/engine/sniper_post_sell_feedback.py`
  - `src/engine/sniper_strength_shadow_feedback.py`
- 산출물:
  1. 예외 분류표(파서/정규화, 외부 I/O, 상태전이/주문)
  2. fail-open/fail-closed 기준표
  3. 로깅 필수 필드(`code`, `stage`, `reason`, `retryable`) 정의
- 완료 기준:
  - “어떤 예외를 어디서 삼킬 수 있는지”가 파일/함수 단위로 명시됨.
  - 매수/매도 의사결정 규칙(임계값/게이트/청산) 불변 확인 문구 포함.
- 검증 명령:
  - `rg -n "except Exception|except:" src/engine src/utils | head -n 300`

### P1-WO-02. 예외 처리 개선용 변경 후보 목록 작성 (코드 수정은 다음 단계)
- 목적:
  - 실제 코드 수정 전에, 변경 후보를 “저위험 순서”로 정렬해 작업 실패 확률을 낮춘다.
- 우선순위:
  1. 리포트/분석 계층 (`sniper_post_sell_feedback.py`, `strategy_position_performance_report.py`)
  2. 보조 동기화/조회 계층 (`sniper_sync.py`, `kiwoom_utils.py`)
  3. 주문/상태전이 핵심 계층 (`sniper_state_handlers.py`, `kiwoom_orders.py`)
- 완료 기준:
  - 각 파일별로 “예외 변경 허용 구간”과 “수정 금지 구간(핵심 상태전이)”이 분리 표기됨.
  - 배포 단위를 작은 PR 단위(파일 1~2개)로 쪼갠 순서안이 존재.

### P1-WO-03. 테스트 보강 오더 (회귀 방어 우선)
- 목적:
  - 성능 개선보다 먼저 “로직 파손 방지”를 테스트로 고정한다.
- 대상 테스트 파일(우선):
  - `src/tests/test_state_handler_fast_signatures.py`
  - `src/tests/test_handle_holding.py`
  - `src/tests/test_kiwoom_orders.py`
  - `src/tests/test_post_sell_feedback.py`
  - `src/tests/test_performance_tuning_report.py`
  - `src/tests/test_pipeline_event_logger.py`
- 추가할 테스트 시나리오:
  1. 외부 API 실패 시: 주문 상태전이 불변 + 에러 원인 로그 기록
  2. 파서/정규화 실패 시: fallback은 허용하되 지표 카운트 누락 없음
  3. post-sell 평가 실패 시: 장후 평가 배치가 중단되지 않고 실패 건수 집계
  4. 민감정보 로그: 토큰/키 패턴이 출력되지 않음
- 완료 기준:
  - 신규 테스트가 “정상경로 + 실패경로”를 모두 포함.
  - 기존 핵심 테스트 대비 수행시간 급증 없음(목표: +15% 이내).
- 검증 명령:
  - `pytest -q src/tests/test_state_handler_fast_signatures.py src/tests/test_handle_holding.py src/tests/test_kiwoom_orders.py`
  - `pytest -q src/tests/test_post_sell_feedback.py src/tests/test_performance_tuning_report.py src/tests/test_pipeline_event_logger.py`

### P1-WO-04. 운영 검증 체크리스트/명령 매핑
- 목적:
  - 코드 변경 전후를 동일 기준으로 비교해 안정성 저하를 즉시 감지한다.
- 체크 항목:
  1. 장중 루프 지연/오류율 악화 여부
  2. 동기화 중복 실행 로그 발생 여부
