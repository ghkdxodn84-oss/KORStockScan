### 1-2. 의사결정 기준 일관성 (검토목적 2)

**판정: 적정**

기대값/순이익 극대화가 §1, §3-1, §7에서 일관되게 명시되어 있다. §3-4의 비교 우선순위(`거래수 → 퍼널 → blocker → 체결품질 → missed_upside → 손익`)는 스캘핑 시스템에서 퍼널 건전성을 손익보다 선행 점검하는 구조로 적절하다.

다만 §3-8에서 "counterfactual 수치는 우선순위 판단 자료로만 쓴다"고 명시한 것은 좋으나, **어떤 과제에서 counterfactual을 생성하는지**가 §8 실행과제표에 표시되어 있지 않다. `buy_recovery_canary`의 paper-fill이 counterfactual 성격인데, 이것이 실현손익 판정과 혼용되지 않도록 §8에 `판정 기준: 실현손익 / counterfactual` 컬럼을 추가하는 것이 안전하다.

## 2. 즉시 시정 항목 (3건)

### [시정 1] §8 실행과제 우선순위 역전 — entry_filter_quality vs position_addition_policy

**위치**: §8 `PA-0422` vs `EFQ-0423`

§8에서 `position_addition_policy` 초안(PA-0422)이 04-22 16:30~17:00에 배치되어 있고, `entry_filter_quality` 착수 재판정(EFQ-0423)이 04-23 POSTCLOSE에 배치되어 있다.

그런데 §1-5에서 "다음 정식 후보는 `entry_filter_quality`다"라고 명시하고, §7 우선 판정에서도 entry_filter가 position_addition_policy보다 상위에 있다. 이전 감사 리뷰(R-4)에서 `entry_filter → holding_exit → position_addition_policy → EOD/NXT` 순서를 권고했고 반영 완료로 확인했다.

그런데 실행과제 일정은 **position_addition_policy 초안(04-22)이 entry_filter 착수 판정(04-23)보다 하루 먼저** 진행된다. "초안 작성"이라 하더라도, 1순위가 아닌 과제에 먼저 시간을 투입하는 것은 원칙과 배치된다. 특히 04-22 16:30~17:00은 `buy_recovery_canary` 1일차 결과를 종합하는 시간대와 겹치므로, 해당 슬롯은 canary 결과 정리에 써야 한다.

**권고조치**:
- `PA-0422`를 04-23 이후로 이전하거나, 04-22 슬롯에서 제거
- 04-22 16:30~17:00은 `buy_recovery_canary` 1일차 종합 판정 + 다음 액션 고정으로 재배정
- `entry_filter_quality` 설계 착수를 04-23 POSTCLOSE에서 04-23 INTRADAY로 앞당기는 것을 검토

### [시정 2] §6 rollback guard — buy_recovery_canary 전용 guard 누락

**위치**: §6 정량 목표와 가드 표

§6에 7개 guard가 정의되어 있으나, 현재 유일한 live canary인 `buy_recovery_canary`에 **직접 대응하는 guard가 `buy_drought_persist` 1개뿐**이다. 이 guard는 "BUY count가 baseline 하위 3분위수 미만이고 blocked_ai_score_share 개선 없음"일 때 canary 유지 금지를 발동하는데, 이는 **canary가 실패할 때의 guard**다.

반대로 **canary가 과도하게 작동할 때의 guard**가 없다. buy_recovery_canary가 WAIT 65~79 구간을 BUY로 전환하면 불량 진입이 증가할 수 있다. 이 경우를 잡는 guard는 `reject_rate`와 `loss_cap`이 범용으로 커버하지만, buy_recovery 특화 지표로 다음이 필요하다:

| Guard 지표 | 발동 조건 | 조치 |
|---|---|---|
| `recovery_false_positive_rate` | canary로 회복된 BUY 중 soft_stop 비율이 baseline normal_only 대비 +5.0%p 이상 | canary OFF, score 재교정 |

이 guard 없이는 "BUY는 회복됐는데 손절도 늘었다"는 상황을 `loss_cap` 발동 전까지 감지하지 못한다.

**권고조치**: §6 표에 `recovery_false_positive_rate` guard 1행 추가.

### [시정 3] §9 일정 — 04-23, 04-24 판정시각이 "POSTCLOSE"로만 기재

**위치**: §9 일정표 마지막 2행

§4에서 "모든 작업일정은 YYYY-MM-DD KST와 Slot/TimeWindow로 고정"이라고 규칙을 명시했다. 그런데 §9에서:

- 04-23: `POSTCLOSE` — TimeWindow 없음
- 04-24: `POSTCLOSE` — TimeWindow 없음

04-22 일정은 모두 `12:00~12:20`, `12:20~12:30`, `15:40~17:00`처럼 TimeWindow가 고정되어 있으나, 04-23~04-24는 Slot만 있고 TimeWindow가 빠져 있다. 이는 §4 자체 규칙 위반이다.

**권고조치**:
- 04-23 POSTCLOSE → `15:20~15:50` (entry_filter_quality 재판정 + A/B preflight)
- 04-24 POSTCLOSE → `15:20~16:00` (주간 통합판정 + A/B 재개 여부)

## 3. 보완 필요 항목 (4건)

### [보완 1] §8 실행과제 — 판정 기준 컬럼 누락

§8 표에 `기대효과`와 `실제효과` 컬럼은 있으나, 각 과제의 **판정 기준이 실현손익인지 counterfactual인지** 구분이 없다. §3-8에서 counterfactual 분리 원칙을 명시한 이상, 실행과제 단위에서도 이를 표기해야 원칙이 실행까지 관통한다.

**권고**: §8 표에 `판정 basis` 컬럼 추가. 값은 `실현손익` / `counterfactual` / `퍼널 지표`.

### [보완 2] §7 Pain Point — "shared 호출 혼입 가능성"의 기대값 영향이 불명확

§7에서 `shared` 호출 혼입의 기대값 영향을 "프롬프트 귀속/성과 귀속 혼선"이라고 기재했는데, 이는 기대값 영향이 아니라 **측정 오염 리스크**다. 기대값 영향은 "shared 호출이 진입 판정을 왜곡하여 불량 진입 N건을 유발했을 가능성"처럼 정량화 방향이 있어야 한다.

**권고**: 기대값 영향을 "shared 혼입 시 진입 판정 귀속 불가 → 불량 진입 원인분석 차단"으로 수정. 04-22 12:20~12:30 강제판정에서 shared 호출 건수와 해당 건의 진입 결과를 확인.

### [보완 3] §4 작업 규칙 — "봇 재실행" 규칙의 범위

§4 "봇 재실행" 규칙에서 "검증/반영에 봇 재실행이 필요하고 권한/안전 조건이 맞으면 AI가 표준 wrapper로 직접 실행"이라고 기재했다. 이 규칙은 합리적이나, **canary ON/OFF도 봇 재실행에 해당하는지**가 불명확하다. canary 플래그 변경이 봇 재실행 없이 hot-reload 가능한지, 아니면 프로세스 재시작이 필요한지에 따라 rollback guard 발동 시 대응 속도가 달라진다.

**권고**: canary ON/OFF가 봇 재실행을 요구하는지, hot-reload 가능한지를 §4에 1행 추가로 명시. rollback guard 발동 시 "canary OFF → 봇 재실행" 소요 시간 예상치도 기재.

### [보완 4] 상태 관리 구조 — 후순위 항목의 판정시각 누락

구버전 중심 문서에서는 별도 상태표에 후순위 항목이 `entry_filter_quality`, `EOD/NXT`, `AI 엔진 A/B`로 나열되어 있었으나, §5 완료 기준에서 "보류/미착수는 다음 판정시각 또는 폐기/코드정리 판정이 명시"되어야 한다고 규정했다. 현행 문서에서는 이 상태 관리가 §8 과제 레지스터의 `상태/일정`으로 통합됐다.

현재 상태:
- `entry_filter_quality`: §8에서 04-23 POSTCLOSE로 판정시각 있음 → 적정
- `EOD/NXT`: 판정시각 없음 → **위반**
- `AI 엔진 A/B`: §8에서 04-24 POSTCLOSE로 판정시각 있음 → 적정

**권고**: `EOD/NXT`에 대해 "04-24 주간 통합판정 시 착수 여부 재판정" 또는 "다음주 이관"을 상태 레지스터에 명시.

## 4. 긍정 평가 (이전 감사 대비 개선점)

이전 리뷰에서 지적한 사항들이 체계적으로 해소되었다. 특히:

1. **shadow 철회 완전 반영**: §4에서 "shadow 금지"를 규칙으로 격상하고 위반 시 처리까지 명시. 이전 문서에서 3회 반복되던 "shadow/counterfactual부터 검증" 문구가 완전히 제거됨
2. **rollback guard 정식 정의**: 이전 리뷰(시정 B)에서 "§9-1 하단 메모에만 수치가 있고 본문에 없다"고 지적한 것이 §6 정식 표로 승격됨. 부호 방향도 "+15.0%p 이상 증가"처럼 명시
3. **용어 범례 도입**: §2에서 fallback 관련 용어 6개와 기준선 용어 3개를 한글 설명과 함께 정의. 이전에 "fallback"이 문맥에 따라 다른 의미로 사용되던 혼동이 해소
4. **관찰 반나절 제한**: §4에서 "live 영향 여부 관찰은 오전/오후 반나절을 넘기지 않음"을 규칙화하고 미관측 시 3개 선택지(관찰축 오류/live 영향 없음/그대로 진행)를 강제. 이전에 shadow 관찰이 2~3일로 늘어나던 문제를 구조적으로 차단
5. **폐기 3중 차단**: §1 영구 폐기 선언 + §2 용어 범례에서 "재개/승격/canary 대상 아님" + §6 `fallback_regression` guard에서 1건이라도 발생 시 즉시 OFF
