# 초단타 종목 스캐너 고도화 제안서
> 작성일: 2026-04-28  
> 대상: 시스템 운영자 / 개발팀  
> 목적: 현재 `scalping_scanner.py`와 키움 REST/WebSocket 가이드를 기준으로, **더 많은 종목**과 **더 품질 좋은 종목**을 동시에 추출하기 위한 스캐너 구조 개선안을 제시한다.

---

## 1. 요약

현재 스캐너는 장중 주기적으로

- `ka10028` 시가대비등락률 상위
- `SniperRadar.find_supernova_targets()`

두 축만 병합하여 후보를 만들고,  
`BOTH > SUPERNOVA > OPEN_TOP` 우선순위 + freshness score로 정렬한 뒤,  
최근 픽 쿨다운과 `max_new_codes` 제한 하에서 웹소켓 감시 등록을 수행한다.

이 구조는 장 초반 강한 갭업/시가대비 강세 종목을 잡는 데는 유효하지만, 다음 한계가 있다.

1. **발굴 소스가 적다.**
2. **발굴과 승격(WS 등록)이 한 단계로 붙어 있다.**
3. **장중 늦게 살아나는 종목을 놓치기 쉽다.**
4. **거래대금/수급/VI/조건검색 기반의 고품질 증거를 충분히 활용하지 않는다.**
5. **고정 쿨다운이 강해서 같은 날 재점화 종목을 재포착하기 어렵다.**

따라서 최적의 방향은 아래 3단 구조다.

> **발굴(source) → 정제(enrich) → 승격(promote)**

즉,

- REST/API 다중 소스로 넓게 모으고
- `ka10095` 등으로 일괄 정제하고
- 최종적으로 `0B + 0D + 0w` 실시간 증거를 붙여 상위 소수만 WS 감시로 올리는 구조가 적합하다.

### 2026-04-29 코드리뷰 반영 상태

- 완료: REST 소스별 예외 격리. `ka10028`, `Supernova`, `ka10032`, `ka10054` 중 일부가 실패해도 남은 소스로 후보 승격을 계속한다.
- 완료: `RankPrev=-1` 같은 순위 없음 sentinel을 보존해 순위 점프 오탐을 줄였다. 가격/거래대금처럼 부호가 방향이 아닌 필드는 별도 양수 정규화로 처리한다.
- 완료: 최근 픽 정리 함수명을 실제 동작에 맞게 `cooldown 이내 유지` 의미로 정리했다.
- 완료: DB 저장 실패는 error 로그로 남기고, DB 저장이 실패한 후보는 cooldown 기억과 WS 등록 대상에서 제외한다.
- 완료: 중복 VI 소스 병합 시 `VIReleaseTime`은 더 최신 시각을 유지한다.
- 남음: DB/WS 이벤트의 완전한 트랜잭션 경계 재설계, `candidate_pool`과 `promoted_watchlist` 분리, `ka10095` 일괄 정제, z-score 합성 점수, 시간대별 소스 분기는 별도 구조 변경으로 유지한다.

## 2. 현재 코드 구조 요약

현재 `scalping_scanner.py`는 다음 흐름으로 동작한다.

1. 장 시간(`09:05 ~ 15:00`) 내에서만 스캔
2. 시간대별 스캔 주기:
   - 09:05~10:30: 120초
   - 14:00~15:00: 120초
   - 그 외: 180초
3. 후보 소스:
   - `get_top_open_fluctuation_ka10028()`
   - `SniperRadar.find_supernova_targets()`
4. 병합 후 `Source`를 `OPEN_TOP / SUPERNOVA / BOTH`로 저장
5. `recent_picks` 기반 재등록 쿨다운 25분
6. `max_new_codes = 12`
7. 선별 후 `COMMAND_WS_REG` 이벤트로 WS 감시 등록

핵심적으로 현재 스캐너는 **소스 다양성은 낮고, 승격 기준은 상대적으로 단순한 편**이다.

## 3. 문제 진단

## 3-1. 발굴 소스 부족
현재 `ka10028`은 장 초반에 잘 보이지만 장중에는 stale leader가 남기 쉽다.  
반면 거래량 급증, 거래대금 급증, VI 발동, 외인/기관 유입, 프로그램 양전환처럼 “늦게 살아나는 종목”은 지금 구조만으로 놓칠 수 있다.

## 3-2. 후보 풀과 감시 풀의 분리 부족
지금은 사실상 “찾은 종목 = 바로 감시 등록 후보”다.  
이 구조는 WS 부하 관리는 쉽지만, 더 많은 종목을 보려면 곧바로 시스템 부담으로 이어진다.

## 3-3. 쿨다운이 고정 시간 기반
현재는 최근 25분 내 뽑힌 종목을 쉬게 한다.  
그러나 같은 날 재점화되는 종목은 장중에도 자주 나온다. 고정 25분 쿨다운은 재상승 종목을 놓치게 만든다.

## 3-4. 품질 정제가 약함
현재 품질 판단은 `Source`, `PriorityScore`, `SpikeRate`, `FluRate`, `CntrStr` 위주다.  
거래대금, 체결강도, 전일대비 거래량 급증률, 5호가 구조, 총잔량, 프로그램 흐름 등 키움이 주는 좋은 정량값을 충분히 활용하지 못하고 있다.

## 4. 권장 구조

## 4-1. 3단 구조로 전환

### 1단: 발굴(Source)
다수의 REST/조건검색/API에서 **넓게 후보를 수집**한다.

### 2단: 정제(Enrich)
후보 코드를 묶어서 `ka10095` 등으로 **일괄 재평가**한다.

### 3단: 승격(Promote)
실시간 `0B + 0D + 0w`와 필요 시 `0H`를 붙여 **상위 소수만 감시 등록**한다.

## 7. 승격(promote) 단계 제안

## 7-1. 실시간 승격 핵심
스캘핑 즉시성은 아래 조합이 핵심이다.

- `0B` 주식체결
- `0D` 주식호가잔량
- `0w` 종목프로그램매매

장전/장후는 보조로:
- `0H` 주식예상체결

즉, **후보를 먼저 많이 만들고**,  
실시간 승격은 `0B + 0D + 0w`로 좁게 가져가는 것이 좋다.

---

## 7-2. 후보 풀 / 감시 풀 분리
권장 구조:

- **Candidate Pool:** 40~80개
- **Promoted Watchlist:** 10~15개

### Candidate Pool
- REST/조건검색 기반으로 넓게 유지
- DB 또는 메모리 캐시에 후보 점수 저장

### Promoted Watchlist
- 상위 점수 종목만 WS 등록
- WS 증거가 약해지면 demote
- 후보 풀에서 다시 승격 가능

---

## 8. 점수화 방식 제안

현재 `Source` 기반 편향 점수를 크게 두는 방식은 유지하되, 비중을 줄이고 증거 누적형으로 바꿔야 한다.

## 8-1. 권장 스코어링 구조
총점 = `source_score + liquidity_score + momentum_score + money_score + structure_score + recency_score`

### source_score
- 여러 소스 중복 출현 여부
- 예:
  - `BOTH`처럼 2개 소스 겹침: 가산
  - `OPEN_TOP` 단독: 가산 적음
  - `VOLUME_SPIKE + VALUE_TOP + COND_REALTIME`: 가장 큼

### liquidity_score
- 거래대금
- 거래회전율
- 호가 총잔량
- 5호가 깊이

### momentum_score
- 등락률
- 거래량 급증률
- 체결강도
- 예상체결량/예상체결가

### money_score
- 프로그램 순매수/증가
- 외인/기관 장중 순매수
- `0w` 양전환

### structure_score
- VWAP 위 여부
- 고점 돌파 시도
- VI 발동 패턴
- orderbook imbalance

### recency_score
- 최근 새롭게 살아난 신호인지
- stale leader인지
- 마지막 후보 진입 시각 대비 freshness

## 9. 쿨다운 및 재포착 로직 개선

현재는 `recent_picks` 25분 고정 쿨다운이다.  
권장안은 **신호 기반 재포착**이다.

### 권장 규칙
- `OPEN_TOP` 단독 포착: 10분
- `VOLUME_SPIKE / VALUE_TOP / COND_REALTIME` 재등장: 즉시 재평가
- source_count 증가 시 쿨다운 무시
- `0w` 양전환 시 재승격 허용

즉, 종목 자체를 막기보다 **동일 신호 에피소드만 쉬게 하는 방식**이 좋다.

## 15. 최종 권고

가장 먼저 할 일은 아래 4가지다.

1. `ka10023`와 `ka10032`를 발굴 소스로 추가한다.
2. `candidate_pool`과 `promoted_watchlist`를 분리한다.
3. 후보 상위 30~50개를 `ka10095`로 일괄 정제한다.
4. 최종 승격은 `0B + 0D + 0w` 실시간 증거가 붙은 종목만 허용한다.

한 줄로 정리하면 아래와 같다.

> **현재 스캐너는 “두 소스에서 바로 감시 등록” 구조라 후보 수와 품질이 동시에 제한된다.**  
> **가장 좋은 개선은 REST 다중 소스로 넓게 발굴하고, `ka10095`로 일괄 정제한 뒤, `0B/0D/0w` 실시간 증거로 상위 소수만 승격하는 3단 구조로 바꾸는 것이다.**
