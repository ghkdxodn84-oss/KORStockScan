# 스캘핑 스캐너 개선 제안

> 키움 REST API 가이드와 현재 `scalping_scanner.py` 코드를 함께 검토한 결과를 정리한 문서입니다.

---

## 핵심 결론

현재 스캐너는 **"가격 모멘텀 한 축 + 거래량 급증 한 축"** 두 갈래로만 돌고 있어서, 스캘핑에서 진짜 알파가 나오는 **거래대금 / 수급 / 변동성 이벤트** 축이 빠져 있습니다.

만약 한 가지만 먼저 도입한다면 **`ka10032` 거래대금상위 + 순위 점프 감지**를 추천합니다. 코드 변경량이 가장 적고, 현재 스캐너의 가장 큰 약점인 "유동성 검증 누락"을 바로 해결합니다.

---

## 1. 현재 스캐너의 진단

### 1-1. 후보 소스가 두 개뿐

코드를 보면 후보 풀이 두 곳에서만 나옵니다.

| 출처 | API | 의미 |
|---|---|---|
| `OPEN_TOP` | `ka10028` | 시가대비 등락률 상위 |
| `SUPERNOVA` | (자체) `find_supernova_targets` | 거래량 급증 추정 (ka10023 계열) |

### 1-2. 점수 함수가 사실상 한 축

`_freshness_score`도 결국 `flu_rate`, `spike_rate`, `cntr_str`만 합산하는 구조라 **"가격이 뛰고 거래량이 늘었다"** 두 신호의 변형들입니다.

### 1-3. 놓치고 있는 케이스

- 등락률은 +3%대 정도지만 **거래대금**이 어제의 5배로 폭발 → 거대 자금 유입 시작점
- **VI 발동** 직후 해제 시점 (5~10분 내 변동성 폭발이 거의 확정적)
- **외국인 + 기관 동시 순매수**가 막 시작된 종목 (수급 전환점)
- 영웅문4에서 직접 만든 정교한 조건식 (매번 코드로 재현하는 건 비효율)

## 2. Tier 1: 후보 풀에 즉시 추가할 4개 API

### 2-1. `ka10032` 거래대금상위요청 — 최우선 추가

스캘핑에서 등락률보다 **거래대금**이 더 중요한 이유는 슬리피지/체결성 때문입니다. 등락률 +20%여도 거래대금 5억이면 스캘핑 진입 자체가 불가능하지만, 등락률 +5%여도 거래대금 1,000억이면 깔끔하게 들고 나올 수 있습니다.

```python
# 호출 예시
{
    "mrkt_tp": "000",      # 전체
    "mang_stk_incls": "0", # 관리종목 미포함
    "stex_tp": "3"         # 통합
}
```

응답 필드 `now_rank`, `pred_rank`를 비교해서 **순위가 급상승한 종목** (예: 어제 50위 → 오늘 5위)을 잡는 것이 핵심입니다. 단순 거래대금 1위는 삼전/하이닉스라 의미 없지만, **순위 점프**는 그 자체로 신호입니다.

### 2-2. `ka10054` 변동성완화장치발동종목요청 — 스캘핑의 꽃

VI는 변동성 자체가 트리거라 스캘핑에 가장 직접적입니다. 결정적인 응답 필드:

| 필드 | 의미 | 활용 |
|---|---|---|
| `motn_pric` | 발동 가격 | 진입 기준선 |
| `virelis_time` | 해제 시각 | **해제 직후 1~3분이 골든타임** |
| `vimotn_cnt` | 당일 VI 발동 횟수 | 3회 이상이면 변동성 폭발 진행 중 |
| `viaplc_tp` | "동적" / "정적" / "동적+정적" | 발동 종류 판별 |

```python
{
    "mrkt_tp": "000",
    "bf_mkrt_tp": "1",          # 정규시장
    "motn_tp": "0",             # 전체 (정적+동적)
    "skip_stk": "111111111",    # 우선주/관리/경고/주의/환기/단기과열/증100/ETF/ETN 모두 제외
    "trde_qty_tp": "1",
    "min_trde_qty": "100000",   # 10만주 이상만
    "trde_prica_tp": "1",
    "min_trde_prica": "1000",   # 10억원 이상 (백만원 단위)
    "motn_drc": "1",            # 상승 발동만
    "stex_tp": "3"
}
```

### 2-3. `ka10171` + `ka10172` 조건검색 — 가장 강력한 leverage

영웅문4에서 직접 만든 복잡한 조건식 (예: "거래량 5일 평균 대비 300% 이상 + 5일 이평선 위 + RSI 70 미만 + 시총 3,000억 이상")을 **웹소켓 한 번 호출로** 받아올 수 있습니다.

이게 강력한 이유는 한국 주식의 미세한 시장 특성 (테마, 호재 패턴, 거래소별 특성)을 본인이 가장 잘 아는 영웅문 GUI에서 튜닝하고, 코드는 결과만 소비하면 되기 때문입니다.

**추천 워크플로우**: 영웅문4에서 다음 3개 조건식을 만들어 두고, 시간대별로 다른 조건을 호출.

1. `스캘핑 후보 A (공격적)` — 장초 변동성 구간용
2. `스캘핑 후보 B (보수적)` — 장중 눌림목 구간용
3. `급등 초입` — 모멘텀 시작점 포착용

```python
# ka10171: 조건식 목록 조회
{ "trnm": "CNSRLST" }

# ka10172: 특정 조건식 일반 조회
{
    "trnm": "CNSRREQ",
    "seq": "4",          # ka10171에서 받은 일련번호
    "search_type": "0",
    "stex_tp": "K",
    "cont_yn": "N",
    "next_key": ""
}
```

### 2-4. `ka90009` 외국인기관매매상위 / `ka10062` 동일순매매순위

스마트머니가 **양쪽에서 동시에** 사들이는 종목은 그날 강한 박스권 또는 추세를 형성할 확률이 높습니다.

`ka10062`의 응답에는 `orgn_nettrde_qty` (기관 순매매수량)와 `for_nettrde_qty` (외인 순매매수량)가 같이 나오므로 **둘 다 양수**인 종목만 골라내면 됩니다.

## 3. Tier 2: 후보를 검증하는 품질 필터

후보 풀이 넓어지면 노이즈도 늘어납니다. 현재 코드는 후보를 모은 뒤 `is_valid_stock` 한 번으로 끝나는데, 거기에 **모멘텀 가속 확인** 단계를 한 층 더 두는 것을 추천합니다.

### 3-1. `ka10046` 체결강도추이시간별 — 가속도 검증

응답에 `cntr_str`, `cntr_str_5min`, `cntr_str_20min`, `cntr_str_60min`이 같이 나옵니다. 이 4개의 **시간 순서 모양**이 진짜 알파입니다.

| 패턴 | 예시 | 판정 |
|---|---|---|
| 60min → 20min → 5min → 현재 단조 증가 | 110 → 120 → 135 → 150 | **가속 매수, 진입 후보** |
| 60min은 높은데 5min이 떨어짐 | 150 → 145 → 130 → 115 | **모멘텀 소진, 회피** |

현재 `_freshness_score`는 단일 시점 `cntr_str`만 보므로 이미 정점을 찍고 떨어지는 종목을 잡는 위험이 있습니다. 이게 후보 → 진입 단계에서 가장 자주 발생하는 손실 원인입니다.

### 3-2. 그 외 보조 필터

- **`ka10018` 고저가근접요청** — 신고가 근접 종목은 돌파 시 거래량 폭발이 거의 확정. 가산점 용도.
- **`ka10024` 거래량갱신요청** — N일 평균 대비 거래량 신고치 갱신. `ka10023` (분 단위) + `ka10024` (일 단위) 조합이 단독보다 노이즈가 훨씬 적음.
- **`ka10052` 거래원순간거래량요청** — 특정 외국계 창구 (메릴린치, 모건스탠리 등) 순간 매수 폭발 감지.

## 4. 코드 통합 제안

현재 `run_scalper`의 구조를 거의 안 바꾸고 다음처럼 확장 가능합니다.

### 4-1. 후보 수집 단계

```python
soaring = kiwoom_utils.get_top_open_fluctuation_ka10028(token, ...)     # 기존
supernova = radar.find_supernova_targets(...)                           # 기존
trade_value = kiwoom_utils.get_top_trade_value_ka10032(token, ...)      # 신규
vi_targets = kiwoom_utils.get_vi_triggered_ka10054(token, ...)          # 신규

condition_results = ws_client.fetch_condition_ka10172(seq=4, ...)       # 신규 (웹소켓)
```

### 4-2. 출처 우선순위 재정의

VI 발동 + 거래대금 폭증 동시 충족이 최강 신호입니다.

```python
SOURCE_PRIORITY = {
    "VI+VALUE+SMART": 0,   # 3개 동시
    "VI+VALUE":       1,
    "CONDITION":      1,   # 사용자가 직접 만든 것이므로 신뢰도 높게
    "VI":             2,   # VI 단독
    "VALUE+SMART":    3,
    "BOTH":           4,   # 기존 (등락률 + 급증)
    "SUPERNOVA":      5,
    "VALUE":          6,
    "OPEN_TOP":       7,
}
```

### 4-3. `_freshness_score`를 z-score 합성으로 교체

단순 합산은 단위가 다른 값들이 섞여 (등락률 +5와 체결강도 150은 비교 불가) 한쪽 신호에 편향됩니다.

```python
def _composite_score(target, market_stats):
    """market_stats는 1분마다 시장 전체 분포로 갱신."""
    z_value          = (target['TradeValue'] - market_stats['value_med']) / market_stats['value_mad']
    z_strength_accel = target['CntrStr5'] - target['CntrStr60']  # 가속도
    z_flu            = (target['FluRate'] - market_stats['flu_med']) / market_stats['flu_mad']

    score = (
        2.0 * z_value           # 거래대금 폭증을 가장 무겁게
        + 1.5 * z_strength_accel # 체결강도 가속
        + 1.0 * z_flu           # 등락률
    )
    if target.get('VI'):
        score += 1000           # VI는 별도 가산
    return score
```

## 5. 시간대별 전략 분리

현재는 모든 시간대에 같은 후보 소스를 쓰지만, 실제로는 시간대별로 효과적인 신호가 다릅니다. `_resolve_scan_interval_sec`처럼 시간대별 분기 로직을 **후보 소스 선택**에도 적용하세요.

| 시간대 | 구간 특성 | 우선 소스 |
|---|---|---|
| 09:05 ~ 09:30 | 장초 변동성 | `ka10054` VI 발동 + `ka10032` 거래대금 순위 점프 |
| 09:30 ~ 14:00 | 눌림목 / 리테스트 | `ka10046` 체결강도 가속 + `ka10062` 외인·기관 동시 순매수 |
| 14:00 ~ 15:00 | 장후 모멘텀 | `ka10032` 1시간 누적 거래대금 급증 + `ka10018` 신고가 근접 |

장초에 등락률 상위는 갭상승만 잔뜩 잡혀서 한물간 정보고, 장중에는 신규 진입 종목보다 이미 한 차례 상승 후 다시 살아나는 종목이 알파입니다.
